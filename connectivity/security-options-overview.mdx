<p>This topic describes the security options available with the AnyNet solution to protect devices and data in transit, particularly over the internet, so that your IoT deployment will work effectively.</p>

<Tip>Contact your Account Manager if you want to set up additional security options for your devices.</Tip>

<h2 id="AboutAnyNetsecurity">About AnyNet security</h2>

<p>The simplified diagram below shows connectivity between IoT devices, the Eseye PoPs and the central systems that devices are communicating with (which might exist in a private or public cloud, PoP, or on-premises servers). </p>

<p><img src="/images/resources/images/connectivity/security overview.png" alt="Security overview" title="Security overview" /> </p>

<p>It also shows how data is secured:</p>

<ul> <li> <span>Device security</span>  <br />Customers are responsible for their device security. For best practice recommendations, including which IoT protocols to use, see <a href="/connectivity/device-configuration" target="_blank">Device configuration best practices</a>. </li> <li> <span>Mobile network security</span>  <br />With a cellular solution, connectivity between devices and the internet is provided by mobile networks. Cellular connectivity is based on GSMA standards and is covered by stringent regulations and quality of service requirements. Cellular networks provide inherent security for data transmission, including encryption of data in transit. </li> <li> <span>Additional AnyNet solution security</span>  <br />Eseye can apply additional security options, such as VPNs and ACL rules, for routing data securely between the Eseye network and the customer or their partner systems. This ensures that data is not intercepted or lost as it transits over the internet, and prevents rogue actors gaining access to devices.  <br />Eseye provides the following security: <ul> <li> <span>Private IP addresses and Network Address Translation (NAT)</span> – ensure IP addresses are hidden from the internet so that they are not used to access a device directly. <br />For more information, see <a href="/connectivity/ip-addresses" target="_blank">About IP addresses</a> and <a href="/connectivity/nat-overview">About Network Address Translation (NAT)</a>. </li> <li> <span>Secure subnets</span> – enable VPN configuration and ACL rules to control and route traffic <br />For more information, see <a href="/connectivity/about-secure-subnets" target="_blank">About secure subnets</a>. <ul> <li> <span>VPNs</span> – enables a high level of security and control for data transfer across the internet. Data is encrypted and access to the VPN is authorised and controlled.  <br />For more information, see <a href="/connectivity/anynet-vpns" target="_blank">Understanding VPNs</a>. </li> <li> <span>ACL</span> – checks network traffic against an Access Control List (ACL) provided by the customer and discards the traffic unless the destination matches one of the destinations in the list. Recommended for routing non-VPN traffic. <br />For more information, see <a href="/connectivity/non-vpn-network-traffic" target="_blank">Routing non-VPN network traffic</a>. <span><strong>Tip: </strong>The public IP addresses for AnyNet PoPs are available for configuring customer-side ACL rules. See <a href="/connectivity/anynet-public-ip-address-ranges">Egress IP addresses</a>.</span> </li> </ul> </li> </ul> </li> </ul>
